## 기본 정보
- 키워드: {keyword}
- 필러 토픽: {pillar}
- 현재 연도: {year}
- 타겟 독자: 한-일 크로스보더 비즈니스 관련 업계 관계자 (마케터, 경영진, 에이전시)
- 톤: 전문적이면서 읽기 쉬운 B2B 톤. 딱딱하지 않되 가볍지도 않게. **반드시 존댓말(~합니다, ~입니다, ~됩니다)로 작성**. 반말(~이다, ~한다, ~된다) 절대 사용 금지.

## 작성 전 필수 단계: 웹 리서치

**글을 쓰기 전에 반드시 web_search 툴을 사용해 일본 시장 현황을 조사하십시오.**

- 한국어와 일본어 양쪽으로 검색 (일본어 키워드가 일본 현지 자료에 더 많이 노출됨)
- 검색 우선순위:
  1. 수치·통계·연도 데이터 (일본 총무성, 경제산업성, Meta/LINE 공식 발표, Statista, 矢野経済研究所 등)
  2. 구체 사례 (한국 브랜드의 일본 진출 케이스, 일본 인플루언서 캠페인 사례)
  3. 최신 트렌드·정책 변화 (광고 규제, 플랫폼 정책, 세대 행동 변화)
- 최대 5회 검색. 검색 쿼리는 전략적으로 선택.

## 핵심 원칙 3가지

### 원칙 1 — 팩트 엄격: 리서치 결과만 사용

모든 숫자/통계/연도별 수치/구체 사례는 **아래 3가지 중 하나로만** 처리하십시오. 4번째 옵션(어림치, 추측, 일반론적 권위)은 절대 금지입니다.

**옵션 A: 리서치 결과 + 출처 명시** — web_search로 확인한 공개 사실만. "일본 총무성 2024년 발표 기준", "Meta 공식 데이터(2024)"처럼 출처와 연도를 수치 바로 앞에 명시. 리서치 결과에 없는 수치는 절대 쓰지 말 것.

**옵션 B: 정성 표현** — 숫자 없이도 의미 전달 가능한 자리: "상당한 비중", "빠르게 성장하는", "주요 플랫폼 중 하나". 리서치에서 찾지 못했지만 방향성이 명확할 때 사용.

**옵션 C: 해당 주장 삭제** — 리서치로 확인되지 않고 정성 표현으로도 대체할 수 없는 주장은 문장 자체를 **삭제**하십시오. 글에서 빠져도 전체 논지에 지장이 없어야 합니다. "모른다고 드러내기"가 아니라 "모르는 건 쓰지 않기"가 원칙입니다.

**절대 금지 표현**:
- "약 30% 수준" — 어림치 금지
- "일반적으로 ~로 알려져 있습니다" — 가짜 권위 금지
- "업계 조사에 따르면" (출처 불명) — 출처 없는 권위 금지
- "2024년 대비 약 2배 성장" (리서치 결과 아님) — 비교 수치 fabrication 금지
- "정확히 4.2%" — 가짜 정밀도 금지
- "[확인 필요]", "[수치 검증]" 같은 플레이스홀더 — 공백을 독자에게 노출 금지
- 과거 연도(2024, 2025 등) 데이터를 {year}년 데이터인 것처럼 표기 금지

### 원칙 2 — 인사이트 밀도: "하나마나한 말" 금지

업계 사람이라면 누구나 아는 자명한 문장, 반대가 성립 안 되는 명제는 공허합니다. 지우십시오.

**금지 표현 예시**:
- "문화적 차이를 이해하는 것이 중요합니다"
- "현지화가 핵심입니다"
- "일본 소비자는 한국과 다릅니다"
- "신중한 접근이 필요합니다"
- "장기적 관점이 중요합니다"

**반박 테스트**: 작성한 문장에 대해 "그 반대가 성립할 수 있는가?"를 자문하십시오.
- 반대도 성립할 수 있으면 → 인사이트 (유지)
- 반대가 말이 안 되면 → 공허 (삭제)

**구체 1개 룰**: 모든 추상 주장 다음에는 리서치로 확인된 구체 예시 1개가 따라와야 합니다. 구체 예시를 리서치에서 찾지 못했으면 **주장 자체를 삭제**하십시오.

### 원칙 3 — 독립적 완결성

글은 그 자체로 완결되어야 합니다. 플레이스홀더·빈칸·"나중에 채울 것" 표시를 남기지 마십시오. 독자는 편집자가 아니라 바쁜 실무자입니다. 편집 작업을 독자에게 외주하는 순간 그 글은 실패입니다.

- `<mark>` 태그 사용 금지 (있는 HTML 허용 태그 목록에서 제거됨)
- "[TODO]", "[확인 필요]", "[사례 삽입]", "[수치 검증]" 같은 공백 마커 금지
- 리서치에서 못 찾은 수치·사례는 정성 표현으로 대체하거나 해당 주장을 삭제

## 형식 팔레트 5종

키워드를 읽고 아래 5개 형식 중 **가장 잘 맞는 하나**를 선택하십시오. 선택한 형식의 골격만 따르고 다른 형식 요소를 섞지 마십시오. 어느 것도 맞지 않으면 **tactical**을 기본값으로 사용하십시오. 선택한 형식은 JSON의 `format_type` 필드에 기록하십시오.

### 형식 1: 개념 설명형 (concept)
- **선택 시그널**: 키워드에 "란?", "이란", "무엇", "유형" 포함 / 정의·분류 성격 / "차이"가 **유형 분류** 맥락일 때 (예: "인플루언서 유형별 차이")
- **골격**:
  1. Quick Answer 허용 — 40-80자 한 문장 정의 (첫 문장)
  2. "왜 지금 이 개념이 필요한가" — 맥락 (200-300자)
  3. "X의 핵심 구성 요소" — 2-4개 요소 짧은 단락 설명 (400-600자)
  4. "비슷해 보이는 Y와 어떻게 다른가" — 정성 비교 (200-400자)
  5. "언제 X를 써야 하고 언제 쓰지 말아야 하나" — 적용 경계 (200-300자)
- **리듬**: 정의 → 왜 → 무엇 → 아닌 것 → 언제
- **금지**: 의문문 H2, 별도 FAQ 섹션
- **길이**: 1,200-1,800자

### 형식 2: 전술 가이드형 (tactical)
- **선택 시그널**: "방법", "설계", "전략", "활용법", "진출", "운영" / 실행 중심
- **골격**:
  1. 짧은 인트로 (2-3문장) — 이 가이드가 해결하는 구체적 실무 문제 명시
  2. "전체 프로세스 한눈에" — 3-5단계 요약 (번호 리스트, 한 줄씩)
  3. 단계별 상세 — 각 단계마다 H2, 150-300자 + 실무 팁 1개
  4. **"이 프로세스가 잘 안 되는 이유"** — 실패 원인 2-3개 (필수 섹션)
  5. 마무리 — "다음 단계로 할 일" 1-2문장
- **리듬**: 문제 → 지도 → 디테일 → 실패 경고 → 다음 행동
- **금지**: Quick Answer 도입, 의문문 H2, 별도 FAQ 섹션
- **길이**: 1,500-2,000자

### 형식 3: 트렌드 브리핑형 (trend)
- **선택 시그널**: "트렌드", "변화", "최근", 연도 포함, "패턴 분석" / 시장 관찰 성격
- **골격**:
  1. 인트로 — "최근 [시장/플랫폼]에서 관찰된 변화" 1-2문장 (Quick Answer 형태 금지)
  2. "무엇이 바뀌었나" — 관찰 2-3개 (각 150-250자, 리서치 근거 필수)
  3. "왜 지금 이런 변화가 일어나는가" — 구조적 원인 해석 (300-400자)
  4. "한국 브랜드에게 무엇을 의미하는가" — 시사점 (200-300자)
  5. 짧은 권고 (1-2문장)
- **리듬**: 관찰 → 해석 → 시사점
- **수치 규칙**: 트렌드형은 수치가 핵심. 리서치로 확인한 수치만 사용하되, 못 찾으면 정성 표현으로 바꾸거나 해당 관찰을 빼고 다른 관찰로 교체.
- **허용**: `title`과 `meta_title`에 {year}년 포함 허용 (이 형식만 예외)
- **금지**: Quick Answer 도입, 의문문 H2, 별도 FAQ 섹션
- **길이**: 1,200-1,800자

### 형식 4: 비교 분석형 (comparative)
- **선택 시그널**: "비교", "vs", "선택 기준", "어느 것이" / **X vs Y 대립 구도**의 "차이" (예: "인스타그램 vs 틱톡 차이")
- **골격**:
  1. 인트로 — "X와 Y 중 어느 쪽을 택할지는 상황에 달려 있습니다" 류 (2-3문장)
  2. HTML `<table>` — 3-5행 × 2-3열 (정성 비교만, 수치는 리서치 확인된 것만)
  3. "X가 맞는 경우" — 조건 2-3개 (각 1-2문장)
  4. "Y가 맞는 경우" — 조건 2-3개 (각 1-2문장)
  5. **"판단이 어려울 때 스스로에게 던질 질문"** — 2-3개 질문 리스트 (필수 섹션, 자가 진단)
  6. 마무리 1-2문장
- **리듬**: 중립 → 테이블 → 각 선택지의 적합성 → 자가 진단
- **금지**: Quick Answer 도입, 본문 의문문 H2 (단 5번 자가 진단 질문은 의문문 허용), 별도 FAQ 섹션
- **길이**: 1,500-2,000자

### 형식 5: 함정/체크리스트형 (pitfall)
- **선택 시그널**: "주의사항", "실수", "흔한", "조심", "체크리스트" / 경고 성격
- **골격**:
  1. Quick Answer 허용 — 40-80자 (예: "X에서 가장 자주 막히는 지점 3가지는…")
  2. "함정 1: [이름]" — 증상 → 왜 빠지나 → 회피법 (250-350자, 완결된 미니 챕터)
  3. "함정 2" 동일
  4. "함정 3" 동일
  5. (선택) 함정 4-5
  6. 체크리스트 — 5-7개 bullet, `<strong>핵심:</strong> 설명` 형식
- **리듬**: 경고 → 증상 → 원인 → 회피
- **금지**: 의문문 H2, 별도 FAQ 섹션
- **길이**: 1,500-2,000자

## 작성 규칙 (공통)

- **글 전체 길이**: 1,200-2,000자 (한국어 기준, HTML 태그 제외). 형식별 상한이 있을 경우 형식 규칙이 우선합니다.
- **존댓말(~합니다/~입니다 체)로 통일**
- "코리너스"를 본문에 1-2회 자연스럽게 언급 (강제 삽입 X)
- 마지막에 CTA 문구: "일본 마케팅 전문가와 상담하고 싶으시다면 코리너스에 문의해 주세요."
- **slug 규칙**: 연도 포함 금지. 영속 URL 원칙. 예: `japan-influencer-marketing-cost-guide` (O) / `japan-influencer-marketing-cost-guide-2026` (X)
- **title 규칙**: 트렌드형에서만 {year}년 포함 허용. 나머지 형식은 연도 없는 영속 제목.
- **meta_title 규칙**: title과 동일 (trend 형식은 title 규칙에 따라 {year}년 포함 가능)
- **HTML 태그**: h2, p, ul, li, table, thead, tbody, tr, th, td, strong, em. 마크다운 아님. `<mark>` 태그 사용 금지.
